# Feature Specification: Design Updates 2026 (News + Site Refresh)

## Status

- Ready for implementation. No blockers. One per-step checkpoint near rollout end (Project Submission removal — confirm with client before executing).

## Overview

Refresh the BluePath public website around a new "News & Events" content model and updated visual treatments for the home, leadership, event, and archive pages. The core deliverable is a wordpress-style news article system that William can publish into immediately, alongside additive Sanity schema work, a Studio Presentation/live-preview wiring, and visual updates across several Gatsby templates.

## Problem Statement

The current site has no real publishing surface. The single Sanity `event` document type is overloaded — it's used for both events (with RSVP) and news/article content, but the schema only carries `name`, `slug`, `eventAt`, `content`, `image`, `description`, which can't support a true editorial article (rich body, pull quotes, inline images, embeds, callouts). William has news content waiting to publish that the current schema and templates cannot represent. In parallel, the home, leadership, and event templates have updated comps from Fika that need to ship, and an unused Project Submission page should be retired.

## Goals

- Ship a publishable News pipeline (Sanity `news` document → `/news/:slug` template) without disturbing existing event content.
- Replace the current `/events` archive with a unified `/news-and-events` chronological wall that supports "Load More" pagination.
- Land visual updates for events, leadership, and the home page below-hero section using existing Sanity data wherever possible.
- Wire Sanity Studio Presentation/live preview so editors see Gatsby-rendered output from inside the Studio.
- Promote `casestudies` to a top-level Studio sidebar item.
- Retire the Project Submission page (with client confirmation) and redirect.

## Core Requirements

### Functional Requirements

**Sanity schema**

- FR-001: A new `news` document type MUST be added with these fields: `title` (string, required), `slug` (slug, source = `title`), `subhead` (string, optional), `publication` (string, optional), `publishedAt` (datetime, required), `heroImage` (image with hotspot), `excerpt` (text), `body` (portable text — see FR-006), `seoDescription` (text). Studio orderings MUST default to `publishedAt desc`. List preview MUST show title + publication + date.
- FR-002: `event.ts` MUST be modified additively only — no field removals, no field renames, no `name → headline` flip. Optional new fields: `publication` (string, optional). All current consumers of `event` (existing site page, RSVP form) MUST continue to work without code changes to form handling.
- FR-003: `team.ts` MUST NOT be modified. The Leadership refresh is visual-only and reuses existing `name`, `role`, `bio`, `image`, `order` fields.
- FR-004: Studio `presentationTool` MUST be configured in `sanity.config.ts` with a `resolve.locations` map per the table in §Proposed Approach. Preview URL MUST be env-controlled (local dev URL vs production).
- FR-005: Studio sidebar MUST promote `casestudies` to a top-level item. `HIDDEN_FROM_AUTO_LIST` MUST be updated so it does not appear twice. The Custom Content sublist MUST no longer include Case Studies.
- FR-006: The `news.body` portable text MUST support all of: bold, italic, headings (H2, H3), bulleted lists, numbered lists, inline links, blockquotes, pull quotes (custom block), inline images with captions (custom block), YouTube embeds (custom block), and bold inline callouts (custom decorator/mark for "Why it matters:" / "The big picture:"-style emphasis). Each custom block/mark MUST have a corresponding Gatsby portable-text serializer.

**Gatsby — routing & pages**

- FR-007: A new `/news-and-events` page MUST query both `allSanityNews` and `allSanityEvent`, merge into a single chronologically-sorted list (desc by `news.publishedAt` and `event.eventAt`), and replace the current `/events` archive functionally and visually.
- FR-008: The `/news-and-events` archive MUST support "Load More" pagination: render the first N items server-side, hide the remainder, and reveal the next N on click. Default page size: 10. State is client-side; no separate paginated routes.
- FR-009: A new `/news/:slug` template MUST be generated by `gatsby-node` `createPage` iterating `allSanityNews`.
- FR-010: The existing `/events/:slug` template MUST be kept (route + page generation unchanged), with only its visual layout updated to match `events-updated-single-page.jpg`. RSVP form behavior (Netlify Forms wiring) MUST remain functionally unchanged — restyling only.
- FR-011: A redirect from `/events` (exact match, not prefix) to `/news-and-events` MUST be added via `gatsby-node` `createRedirect` or `netlify.toml`. `/events/:slug` detail routes MUST continue to work.
- FR-012: Header navigation "News & Events" link MUST point to `/news-and-events`.

**Gatsby — visual updates**

- FR-013: The `/news-and-events` archive MUST use the new visual style: white background, BluePath-blue Lora serif headlines, Inter sans body, and per-row layout (thumbnail left; date + publication/location label + headline + subhead + excerpt + CTA right). News CTA: `CONTINUE READING` → `/news/:slug` (orange on hover). Event CTA: `VIEW EVENT DETAILS` → `/events/:slug`.
- FR-014: The `/news/:slug` template MUST be a centered single-column article (~720px reading column) with: brand mark, Lora headline, hero image (full column width), date stamp, then the rendered portable-text body. All custom blocks/marks from FR-006 MUST render correctly.
- FR-015: The `/events/:slug` template MUST adopt the two-column desktop layout (left: vertical badge image; right: date, location label, Lora headline, portable-text body, RSVP form) per `events-updated-single-page.jpg`. Mobile breakpoint behavior MUST follow the existing CSS breakpoints in the codebase.
- FR-016: The Leadership page MUST be refreshed visually (Lora names, Inter role caps, side-rail "LEADERSHIP" label, blue-on-white). Click on a photo or name MUST expand an inline bio panel (rendered from existing `team.bio` portable text) that pushes content down. No schema change.
- FR-017: The home page below-hero section MUST be implemented per `home-first-section.jpg` as the Gatsby renderer for the existing `TOC` content type (referenced from a `homesections` document via `homesections.contentType`). No Sanity schema changes are required — the editor wires the section in `homesections` pointing at the existing `TOC` `contentType` document; the new visual treatment is implemented entirely in the Gatsby renderer for that variant. Layout: large Lora headline, four hexagonal/diamond image tiles, two paragraphs of body copy, then a "NATIONAL PROJECTS" strip (blue side label + horizontal state tiles with state caps + amount + icon). Final headline copy and body copy MUST match the values quoted in the rough plan.
- FR-018: The Project Submission page and its nav entry MUST be deleted, with a redirect added to `/connect` (contact page). This step is gated on explicit client confirmation that submissions are no longer being used (see Open Questions Q1).

**Typography & global**

- FR-019: Lora (serif, headings) and Inter (sans, body and link/nav text) MUST be loaded once via `gatsby-browser`/`gatsby-ssr`. Both files MUST be kept in sync per the project's CLAUDE.md rule. No other serif/sans pairs may remain.
- FR-020: New listing pages (the `/news-and-events` archive) MUST use white background with BluePath-blue text, departing from the existing site's blue-bg/white-text pattern.

### Non-Functional Requirements

- NFR-001: All new and updated pages MUST follow the existing CSS breakpoints. Mobile layouts MUST follow the comp images (no separate mobile design discovery is in scope).
- NFR-002: Existing event content and the existing RSVP form submissions pipeline MUST continue to function during and after deployment. No data migration is performed.
- NFR-003: News articles MUST emit basic SEO metadata: `<title>`, meta description from `seoDescription`, OpenGraph (`og:title`, `og:description`, `og:image`, `og:type=article`), and Twitter card. Structured-data (`Article`) is out of scope for this cycle.
- NFR-004: The Studio Presentation preview URL MUST be env-controlled — local Gatsby URL during development, production URL on deployed Studio. Project ID `qwwmf79r` and dataset `production` MUST continue to be the only targets.
- NFR-005: Build/dev tooling MUST continue using `pnpm` and the pinned Gatsby pre-release (`5.14.0-next.4`). No package manager or Gatsby version changes.

## Proposed Approach

### 1. Sanity (`bluepath-sanity/`)

**1a. New `news.ts` document type.** Field shape exactly as in FR-001. The body is a custom `block` array distinct from the `event.content` shape — it carries the full editorial range (FR-006). Custom block types: `pullQuote`, `imageWithCaption`, `youtube`. Custom decorator/mark: `callout` (or styled bold variant) for inline "Why it matters:"-style emphasis.

**1b. `event.ts` — additive only.** Keep all existing fields. Add optional `publication` field (used for the small location/source label in the new event single-page comp). Add Studio `orderings` by `eventAt desc` and refine list preview. No field renames, no removals.

**1c. `team.ts`.** No schema changes.

**1d. Studio Presentation.** Configure `presentationTool` in `sanity.config.ts` with `resolve.locations` per:

| Schema type            | Studio preview URL                                      |
| ---------------------- | ------------------------------------------------------- |
| `news`                 | `/news/:slug`                                           |
| `event`                | `/events/:slug`                                         |
| `page`                 | `/${slug}` (renders the route; not block-by-block edit) |
| `navigation`           | `/`                                                     |
| `homevideo`            | `/`                                                     |
| `homesections`         | `/`                                                     |
| `carousel`             | `/`                                                     |
| `carouselLink`         | `/`                                                     |
| `casestudies`          | `/projects/`                                            |
| `team`                 | `/leadership`                                           |
| `impact`               | `/impact`                                               |
| `carbonoffsets`        | `/impact`                                               |
| `carbonoffsetstabs`    | `/impact`                                               |
| `carbonoffsetsHexagon` | `/impact`                                               |
| `address`              | `/connect`                                              |

`siteSettings` and `contentType` are intentionally omitted — they have no meaningful single-page preview.

**1e. Sidebar restructure (`sidebar.tsx`).** Promote `casestudies` to a top-level item. Final order: Pages → News → Events → Case Studies → Navigation → (auto-listed remaining) → Custom Content (Home Video, Home Carousel, Home Sections, Team Members, Impact, Carbon Offsets Data, Company Addresses) → Global Settings. Update `HIDDEN_FROM_AUTO_LIST` to prevent `casestudies` from appearing twice.

### 2. Gatsby (`bluepath-gatsby-ts/`)

**Routing.**

- New: `src/pages/news-and-events.tsx` (mixed archive, with Load More).
- New: `src/templates/NewsArticle.tsx` (article template).
- Updated visually only: `src/templates/Event.tsx`, `src/pages/leadership.tsx` (or equivalent), home page below-hero section.
- Removed: existing `src/pages/events.tsx` (after the new archive ships) and Project Submission page (gated on client confirmation).
- Redirects: `/events` → `/news-and-events` (exact match); Project Submission path → `/connect`.

**Mixed archive (`/news-and-events`).** GROQ/GraphQL queries `allSanityNews` and `allSanityEvent`, merges into a single array, sorts desc by the appropriate date field per item, then renders all items but only displays the first 10. A "LOAD MORE" button reveals the next 10 (client-side state). All items are prefetched at build time — pagination is purely a presentation concern.

**News article template.** Centered ~720px column. Renders title, hero image, date, then portable-text body via custom serializers for each block type from FR-006.

**Event template.** Two-column desktop layout, restyled inputs for the existing Netlify Forms RSVP. The form's `data-netlify` attribute, hidden `form-name` input, and POST behavior are untouched.

**Home below-hero section.** Implemented as the Gatsby renderer for the existing `TOC` content type. The `homesections` schema and the `contentType` document are unchanged — an editor adds (or already has) a `homesections` entry whose `contentType` reference points at the `TOC` `contentType` document, and the new layout (Lora headline, four hex tiles, two body paragraphs, NATIONAL PROJECTS strip with state tiles) is implemented purely on the Gatsby render side for that variant. No new schema, no new doc type.

**Leadership.** Visual refresh + click-to-expand bio panel using existing portable-text serializers (subset of the news serializers — bold, italic, headings, links, lists).

**Typography.** Verify Lora + Inter loaded exactly once via `gatsby-browser` and `gatsby-ssr` (mirrored). Remove any other serif/sans loads.

## User Stories

### Story 1: Editor publishes a news article

**As an** editor (William)
**I want to** create a news article in Sanity Studio with a hero image, headline, subhead, publication label, date, and richly-formatted body
**So that** the article appears at `/news/:slug` and on the `/news-and-events` archive without engineering intervention.

**Acceptance Criteria:**

- [ ] Given a published `news` document, when the site rebuilds, then the article is reachable at `/news/<slug>`.
- [ ] Given a published `news` document, when an editor opens the document in Studio, then a Presentation preview link points at the live `/news/:slug`.
- [ ] Given an article body containing a pull quote, inline captioned image, YouTube embed, blockquote, bullet list, numbered list, and bold inline callout, when the article renders, then each block displays correctly with its dedicated styling.
- [ ] Given a news article and an event with adjacent dates, when the user visits `/news-and-events`, then both items appear in the same list, sorted desc by date.

### Story 2: Visitor browses News & Events

**As a** site visitor
**I want to** browse the latest news and upcoming/past events in one place
**So that** I can discover BluePath's recent activity without bouncing between pages.

**Acceptance Criteria:**

- [ ] Given the archive page loads, when there are more than 10 combined items, then only the first 10 are visible and a "LOAD MORE" button is shown.
- [ ] Given the user clicks "LOAD MORE", when the next batch is revealed, then the page does not navigate and the button hides once all items are visible.
- [ ] Given the user clicks `CONTINUE READING` on a news row, when the link resolves, then they land on `/news/:slug`.
- [ ] Given the user clicks `VIEW EVENT DETAILS` on an event row, when the link resolves, then they land on `/events/:slug` with the existing event template (visually updated).

### Story 3: Editor uses Studio live preview

**As an** editor
**I want to** see how my changes render on the live Gatsby site from inside the Studio
**So that** I can iterate on copy/imagery without context-switching.

**Acceptance Criteria:**

- [ ] Given an editor opens a `news`, `event`, `team`, or `casestudies` document in Studio, when the Presentation panel loads, then it shows the corresponding live page.
- [ ] Given the Studio runs in local dev, when the preview loads, then it points at the local Gatsby dev URL (per env config).
- [ ] Given the Studio runs in production, when the preview loads, then it points at the production URL.

### Story 4: Old `/events` URL still works

**As a** visitor following an old shared link or bookmark to `/events`
**I want to** land on the new archive
**So that** the link doesn't 404.

**Acceptance Criteria:**

- [ ] Given a request to `/events` (exact path), when the redirect resolves, then the user lands on `/news-and-events`.
- [ ] Given a request to `/events/:slug`, when the route resolves, then the existing event detail page renders normally (no redirect).

## Technical Constraints

- The two sub-projects (`bluepath-gatsby-ts/`, `bluepath-sanity/`) MUST be developed using `pnpm`; do not use npm or yarn.
- Gatsby is pinned to `5.14.0-next.4` — do not upgrade.
- Sanity project ID `qwwmf79r` and dataset `production` are the only targets.
- `bluepath-gatsby-ts/src/gatsby-types.d.ts` is auto-generated — never edit manually.
- `gatsby-browser.tsx` and `gatsby-ssr.tsx` both wrap pages in `Layout`; any layout-wrapper change must be mirrored in both files.
- `event.ts` is in production use by editors; modifications must be strictly additive (no field removal, no rename, no type changes on existing fields).
- The RSVP form on `/events/:slug` uses Netlify Forms; the `data-netlify`, `name`, and hidden `form-name` input wiring must be preserved exactly.

## Scope Boundaries

### What This Is NOT

- **Not a data migration.** No existing `event` documents are converted to `news`. Editors create new `news` documents from scratch.
- **Not a Gatsby upgrade.** Pre-release pin stays.
- **Not a `team.ts` redesign.** Leadership uses existing fields; the change is purely presentational + a click-to-expand interaction.
- **Not a `/news` standalone index.** Only the unified `/news-and-events` archive exists. A news-only index can be added later if requested.
- **Not a structured-data SEO push.** Basic OG/Twitter metadata only; `Article` schema and JSON-LD are deferred.
- **Not a redesign of `casestudies`, `impact`, or `connect` pages** — those are only touched insofar as the Studio Presentation table points at them.
- **Not a tablet/mobile redesign discovery effort.** Existing CSS breakpoints + comp images are the source of truth; if a mobile layout question is not answered by either, raise it during implementation rather than improvising.
- **Not an export of historical Project Submission form data.** The current flow sends submissions to email; there is no stored backlog to migrate.

### Assumptions

- Lora and Inter are licensed for use on BluePath domains (or are Google Fonts).
- William's pending news content fits the new `news` schema (title, subhead, publication, hero image, body) without needing additional fields.
- The home below-hero section corresponds to the existing `TOC` `contentType` document (referenced from a `homesections` entry). No `homesections` or `contentType` schema changes are needed; the comp is realized entirely in the Gatsby renderer for that content-type variant.
- Editors will use the new sidebar structure once shipped — no parallel editing surface needed.
- Comps are desktop-first; existing CSS breakpoints handle the responsive transitions correctly.
- The client (Jocelyn) will give a clear yes/no on Project Submission removal at step 7. Until then, the page remains live.

## Critical Review

### Why This Approach

- **Splitting `news` from `event`** is the cheapest correct fix: the two content shapes are genuinely different (article vs. RSVP-able event), and forcing them into one document has been the underlying cause of the missing editorial body capability. Splitting now (without migration, since there is no historical news content) is the lowest-risk path.
- **Additive-only changes to `event.ts`** mean the live editing flow and the RSVP form keep working through the entire rollout. There is no big-bang cutover.
- **Single chronological archive with Load More** matches the current production behavior visually (a single wall of items) — verified at https://bluepathfinance.com/events/ — while fixing its main UX flaw (everything dumped on one page with no progressive disclosure).
- **Visual-only Leadership refresh** avoids touching `team.ts`, the highest-risk schema to change since it carries multiple stakeholder photos and bios.
- **`/events/:slug` template kept (visually updated)** preserves Netlify Forms wiring and avoids regression on RSVP — restyle only.
- **Studio Presentation** was added to scope only because it materially improves editor velocity for William's incoming articles (see User Story 3) and is in-cycle work that benefits all subsequent content updates, not just this one.

### Weak Spots / Trade-offs

- **Custom portable-text blocks (FR-006) are real implementation cost.** Five custom block/mark types each need a Sanity schema entry, a Studio input UI, and a Gatsby renderer. This is the largest single chunk of work in step 3 of the rollout and should be scoped accordingly.
- **No editor enforcement that "news goes in `news`, events go in `event`".** An editor could in principle still mis-file content (an event in `news` or a press release in `event`). Mitigation: the sidebar restructure (§1e) groups them clearly, and the Studio document descriptions should be updated to reinforce the distinction.
- **Mixed-list sort interleaves upcoming events with old news.** This is the explicit user choice (matches current behavior at `bluepathfinance.com/events/`), but it means an upcoming event published months ago could sit lower than a recent press release, and a far-future event could sit at the top above today's news. Acceptable per design direction, flagged here for future revisit.
- **Load More is client-side state**, so deep-linking to "page 3" is not supported. Acceptable given small dataset (~20 items today) and that the alternative (paginated routes) adds build-time complexity for marginal SEO benefit on an archive page.
- **Studio Presentation requires env-controlled URLs.** Local dev needs `GATSBY_DEV_URL` (or similar) wired through Sanity Studio's runtime — there's a small footgun if the env var is missing, the preview will silently point at the wrong target.
- **Home below-hero is bound to the existing `TOC` content type.** No schema risk, but the renderer for the `TOC` variant is the only place the layout lives — if a future content-type variant needs similar tile/strip primitives, refactor those out of the `TOC` renderer rather than copying.

## Alternatives Considered

### Alternative 1: Keep `event.ts` as the single content type, expand its body shape

- **Description**: Instead of a new `news` doc type, expand `event.content` to support all the article-grade blocks, then route by date (`eventAt` past = news, future = event) or by a new boolean flag.
- **Why rejected**: Conflates two semantically different content shapes (RSVP-able event vs. article), forces editors to ignore irrelevant fields per item, and makes the listing page logic more complex. The cost of a new doc type (one schema file, one template) is lower than the long-term cost of the conflation.

### Alternative 2: Paginated news/events routes (`/news-and-events/page/2`)

- **Description**: Build `gatsby-paginate`-style numbered pages instead of client-side Load More.
- **Why rejected**: With ~20 items today, the SEO and build-time cost outweighs the benefit. Load More matches modern editorial-archive UX expectations and is materially simpler.

### Alternative 3: Two separate archive pages (`/news` and `/events`)

- **Description**: Keep `/events` for events, add `/news` for articles, keep them separate.
- **Why rejected**: The current production behavior is a single chronological wall (verified live), and the design direction explicitly calls for a unified `/news-and-events` archive. Splitting would be a regression in stakeholder intent.

### Alternative 4: Migrate existing `event` docs into a unified `content` doc type

- **Description**: One document type with a `kind` discriminator field (`"news" | "event"`).
- **Why rejected**: Requires a migration of live data — explicitly out of scope per the rough plan ("no data migration needed"). The split-doc-types approach achieves the same separation without touching live records.

### Alternative 5: Defer Studio Presentation to a later cycle

- **Description**: Skip §1d entirely — ship just the schemas and templates.
- **Why rejected**: Editor velocity for William's incoming articles is a stated goal, and Presentation is comparatively small work (one config block per doc type). Deferring it would mean editors blindly publish-and-refresh during the period when the most new content is being created.
